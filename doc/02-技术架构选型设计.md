# 三审制稿件工作台 - 技术架构选型设计（MVP）

本文基于《01-需求详细设计》与原始PDF需求，给出MVP阶段可落地的技术架构选型与设计约束，目标是：快速交付、稳定运行、便于后续扩展。

---

## 1. 架构目标与约束

## 1.1 架构目标
- 支撑“三审制”完整业务闭环：创建、提交、一/二/三审、退回、重提、完成。
- 明确状态机流转与审计流水，保证流程可追溯、可验证。
- 在MVP阶段保持实现简洁，降低开发与测试成本。

## 1.2 约束条件
- 遵循现有项目基础：`Spring Boot 3.2.4`、`Java 17`、`MyBatis-Plus`、`MySQL`。
- 先实现单体应用（Monolith），不引入微服务拆分复杂度。
- 非MVP能力（RBAC、消息通知、AI等）预留扩展点，不提前过度设计。

---

## 2. 技术选型与取舍

## 2.1 后端选型

| 维度 | 选型 | 选型理由 |
|------|------|----------|
| 应用框架 | Spring Boot 3.2.4 | 成熟稳定，生态完整，适合快速构建REST服务 |
| 语言版本 | Java 17 | LTS版本，兼容性与性能平衡，符合当前工程配置 |
| Web层 | spring-boot-starter-web | 满足MVP接口需求，开发效率高 |
| ORM/持久层 | MyBatis-Plus | 在保留MyBatis灵活SQL能力的同时，减少基础CRUD样板代码，提高MVP开发效率 |
| 监控 | Spring Actuator | 提供健康检查与基础可观测能力 |
| 构建工具 | Maven | 与当前仓库一致，便于CI集成 |

## 2.2 数据库选型

| 维度 | 选型 | 选型理由 |
|------|------|----------|
| 数据库 | MySQL 8.x（兼容5.7语法） | 关系模型适合状态机与审计流水，事务支持完善 |
| 事务策略 | 本地事务（单库） | 保障“状态变更 + 流水记录”原子性 |
| 索引策略 | 按状态、更新时间、稿件ID建索引 | 覆盖列表查询与详情流水查询热点 |

## 2.3 前端选型

| 维度 | 选型 | 选型理由 |
|------|------|----------|
| 形态 | 原生HTML + 原生JavaScript（静态资源 + REST） | MVP开发快、依赖少、部署简单，不引入额外前端框架复杂度 |
| 页面结构 | 列表页 + 编辑页 + 详情/审核页 | 完整覆盖PDF要求的页面能力 |
| 交互策略 | “前端限制 + 后端兜底”双重校验 | 既保证体验，又保证业务安全性 |

---

## 3. 总体架构设计

## 3.1 架构风格

采用“前后端分离但同仓部署”的单体架构：
- 前端静态页面由Spring Boot提供或独立托管；
- 前端通过HTTP调用后端REST接口；
- 后端统一处理状态机规则、审核动作和流水记录；
- 数据持久化到MySQL。

## 3.1.1 前端架构说明（MVP）

- 采用多页面或轻量SPA均可，但技术实现限定为原生`HTML/CSS/JavaScript`。
- 前端通过`fetch/XMLHttpRequest`调用后端REST接口，不引入React/Vue等框架。
- 页面按业务拆分为列表、编辑、详情/审核三个主视图，状态驱动按钮显隐与可操作性。

## 3.2 逻辑分层

| 层级 | 主要职责 | 关键约束 |
|------|----------|----------|
| Controller层 | 接收请求、参数校验、返回统一响应 | 不承载复杂业务规则 |
| Service层 | 状态机流转、业务编排、事务控制 | 所有状态变更必须从此层进入 |
| Repository/Mapper层 | SQL执行、对象映射 | 查询与更新语义清晰，避免隐式副作用 |
| Domain模型层 | 稿件、审核记录、状态枚举 | 状态与动作定义集中管理 |

## 3.3 关键交互链路

- 创建/编辑链路：页面 -> 稿件接口 -> 稿件表更新。
- 审核链路：页面 -> 审批接口 -> 状态机校验 -> 状态更新 + 流水写入（同事务）。
- 查询链路：页面 -> 列表/详情/流水接口 -> 组合展示。

## 4. 状态机技术落地方案

## 4.1 实现原则
- 使用“显式状态转移表”实现状态机，不依赖隐式if拼接。
- 任意写操作前均校验“当前状态 + 目标动作”是否合法。
- 非法流转统一返回业务错误码，便于前端处理提示。

## 4.2 关键规则
- 退回动作仅允许在`PENDING_1ST/2ND/3RD`执行，且意见必填。
- 退回进入`REJECTED`时必须落库`reject_level`。
- 重新提交时根据`reject_level`回到上一关（1->DRAFT, 2->PENDING_1ST, 3->PENDING_2ND）。
- `COMPLETED`视为终态，仅允许查询。

## 4.3 一致性保障
- 状态更新与审核流水记录放在同一事务。
- 若流水写入失败，状态更新回滚，避免数据不一致。

## 4.4 审批引擎版本演进（v1 -> v2）

### v1（当前MVP）
- 采用硬编码三节点：`PENDING_1ST`、`PENDING_2ND`、`PENDING_3RD`。
- 通过固定状态枚举和固定流转规则实现审批。
- 优点：实现快、测试面清晰；缺点：新增节点需改代码与测试。

### v2（规划版本）
- 升级为“节点驱动”审批：节点与事件由配置驱动，而非写死在代码中。
- 前端可配置并维护：
  - 节点定义（如：一审、二审、三审、终审）
  - 事件定义（如：提交、通过、退回、重提）
  - 节点间事件流转关系（from -> event -> to）
- 后端负责配置校验、权限校验、状态机执行与审计落库，前端配置不直接绕过后端规则。

### 兼容策略
- 对外接口优先保持`/api/v1`不变，避免前端一次性重构。
- 新增节点时优先通过配置扩展，减少后端业务代码改动。
- 审核流水增加节点编码/名称字段，保证历史记录在v1/v2期间可追溯。

## 5. 可扩展性设计

## 5.1 面向二期扩展点
- 权限扩展：在Service入口增加角色校验拦截（作者、一审员、二审员、三审员）。
- 通知扩展：在审核动作成功后发布领域事件，后续接消息队列或站内信。
- 发布扩展：在`COMPLETED`后触发可配置Hook，模拟外部平台发布。
- 前端框架扩展：后续可平滑升级到Vue/React，实现组件化与更复杂状态管理。

## 5.2 兼容扩展原则
- 接口版本化预留（如`/api/v1/...`），减少后续变更冲击。
- 状态枚举集中定义，新增状态需同步转移表与测试用例。
- 审核流水模型保持通用字段，支持后续扩展操作者、角色、来源系统。

---

## 6. 环境与部署建议

## 6.1 环境划分
- `dev`：本地开发联调。
- `test`：功能回归与验收测试。
- `prod`：生产运行环境。

## 6.2 配置策略
- 使用`application-{profile}.yml`按环境隔离配置。
- 数据库连接、账号密码等敏感信息通过环境变量或外部配置注入，不入仓库。
- 统一时区建议`Asia/Shanghai`，避免审核时间展示偏差。

## 6.3 运维可观测性
- 开启Actuator健康检查（至少`/actuator/health`）。
- 关键业务日志记录稿件ID、动作、前后状态与操作结果，便于排障审计。

---

## 7. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 状态流转规则实现分散 | 易出现逻辑漏洞 | 统一收敛到Service状态机组件 |
| 前端按钮限制与后端规则不一致 | 用户体验混乱、出现误操作 | 以前端状态矩阵为展示基准，后端规则为唯一真相 |
| 状态与流水写入不一致 | 审计链断裂 | 强制事务一致性 + 异常回滚 |
| 二期扩展影响MVP代码 | 返工成本高 | 预留接口版本、事件扩展点、统一枚举管理 |

---

## 8. 本文结论

MVP阶段采用“Spring Boot单体 + MyBatis-Plus + MySQL + 原生HTML/JS前端页面”的技术架构，能够以较低复杂度满足PDF要求的业务闭环与状态机规则；并通过统一状态机实现和事务化流水记录，为后续权限、通知、发布、前端框架升级等能力扩展保留稳定演进路径。
