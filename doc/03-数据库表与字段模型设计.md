# 三审制稿件工作台 - 数据库表与字段模型设计（MVP）

本文基于《01-需求详细设计》《02-技术架构选型设计》与现有SQL草案，给出MVP阶段可直接落地的数据库模型设计。目标是确保审批状态机、审核流水、查询性能三者一致。

---

## 1. 设计目标与原则

## 1.1 设计目标
- 支撑三审制状态机的全部流转规则。
- 确保“状态变更 + 审核流水”可追溯、可审计。
- 满足MVP列表、详情、审核操作的查询与写入性能。

## 1.2 设计原则
- 单库单事务：保证状态与流水一致性。
- 字段语义清晰：状态码、动作码、退回关卡统一定义。
- 对MyBatis-Plus友好：命名规范、主键策略、时间字段统一。

## 3. 核心实体关系

## 3.1 ER关系说明
- 一篇稿件（`ms_manuscript`）对应多条审核/操作记录（`ms_review_record`）。
- 关系类型：`ms_manuscript (1) -> (N) ms_review_record`。
- 外键关系：`ms_review_record.manuscript_id`引用`ms_manuscript.id`，随稿件级联删除。

## 3.2 业务语义补充
- 稿件当前状态以`ms_manuscript.status`为准。
- 审核轨迹以`ms_review_record`为准，可完整回放历史动作。

---

## 4. 表结构设计

## 4.1 稿件主表 `ms_manuscript`

### 4.1.1 建表目的
存储稿件正文与当前状态，是审批流状态机的“当前快照”。

### 4.1.2 字段定义

| 字段名 | 类型 | 非空 | 默认值 | 说明 |
|--------|------|------|--------|------|
| `id` | `BIGINT UNSIGNED` | 是 | 自增 | 主键 |
| `title` | `VARCHAR(500)` | 是 | - | 稿件标题 |
| `body` | `MEDIUMTEXT` | 是 | - | 稿件正文 |
| `status` | `VARCHAR(32)` | 是 | - | 当前状态码（见状态枚举） |
| `reject_review_level` | `TINYINT UNSIGNED` | 否 | `NULL` | 退回时审核关卡（1/2/3），仅`REJECTED`时有意义 |
| `created_at` | `DATETIME(3)` | 是 | `CURRENT_TIMESTAMP(3)` | 创建时间 |
| `updated_at` | `DATETIME(3)` | 是 | `CURRENT_TIMESTAMP(3)` + ON UPDATE | 更新时间 |

### 4.1.3 索引与约束
- 主键：`PRIMARY KEY (id)`
- 二级索引：`idx_ms_manuscript_status_updated(status, updated_at)`，用于列表按状态筛选与时间排序。

### 4.1.4 字段规则
- `status`必须是约定枚举之一，禁止自由文本。
- `reject_review_level`建议由应用层控制：
  - 当`status = REJECTED`时，值应为`1/2/3`；
  - 其他状态下可置`NULL`（推荐）或保留最近值（需统一口径）。

---

## 4.2 审核记录表 `ms_review_record`

### 4.2.1 建表目的
记录提交、通过、退回、重提、保存草稿等动作，作为审计与问题排查依据。

### 4.2.2 字段定义

| 字段名 | 类型 | 非空 | 默认值 | 说明 |
|--------|------|------|--------|------|
| `id` | `BIGINT UNSIGNED` | 是 | 自增 | 主键 |
| `manuscript_id` | `BIGINT UNSIGNED` | 是 | - | 关联稿件ID |
| `action` | `VARCHAR(32)` | 是 | - | 动作码（见动作枚举） |
| `from_status` | `VARCHAR(32)` | 否 | `NULL` | 变更前状态 |
| `to_status` | `VARCHAR(32)` | 否 | `NULL` | 变更后状态 |
| `opinion` | `VARCHAR(2000)` | 否 | `NULL` | 审核意见，`REJECT`时必填 |
| `reject_level` | `TINYINT UNSIGNED` | 否 | `NULL` | 本次退回关卡（1/2/3） |
| `operator_id` | `BIGINT UNSIGNED` | 否 | `NULL` | 操作人ID（MVP可固定） |
| `operator_name` | `VARCHAR(64)` | 否 | `NULL` | 操作人展示名 |
| `created_at` | `DATETIME(3)` | 是 | `CURRENT_TIMESTAMP(3)` | 操作时间 |

### 4.2.3 索引与约束
- 主键：`PRIMARY KEY (id)`
- 查询索引：`idx_ms_review_manuscript_time(manuscript_id, created_at)`，用于详情页按时间展示流水。
- 外键约束：`FOREIGN KEY (manuscript_id) REFERENCES ms_manuscript(id) ON DELETE CASCADE`

### 4.2.4 字段规则
- `REJECT`动作时：`opinion`必填，`reject_level`必填。
- 其他动作：`opinion`可空，`reject_level`通常为`NULL`。

---

## 5. 枚举模型设计

## 5.1 状态枚举（`status`）

| 状态码 | 中文语义 | 说明 |
|--------|----------|------|
| `DRAFT` | 草稿 | 可编辑、可提交 |
| `PENDING_1ST` | 待一审 | 一审通过/退回 |
| `PENDING_2ND` | 待二审 | 二审通过/退回 |
| `PENDING_3RD` | 待三审 | 三审通过/退回 |
| `REJECTED` | 已退回 | 可编辑、可重提 |
| `COMPLETED` | 已完成 | 终态，只读 |

## 5.2 动作枚举（`action`）

| 动作码 | 触发场景 | 说明 |
|--------|----------|------|
| `SAVE_DRAFT` | 草稿保存 | 不改变审核阶段 |
| `SUBMIT` | 草稿提交 | `DRAFT -> PENDING_1ST` |
| `APPROVE` | 审核通过 | 一/二/三审逐级推进 |
| `REJECT` | 审核退回 | 任一待审态 -> `REJECTED` |
| `RESUBMIT` | 退回重提 | 按`reject_level`回上一关 |

---

## 6. 状态机与字段联动规则

| 业务动作 | 稿件表变化 | 流水表变化 |
|----------|------------|------------|
| 保存草稿 | `status`保持`DRAFT`，更新`updated_at` | 写入`SAVE_DRAFT` |
| 提交审核 | `DRAFT -> PENDING_1ST` | 写入`SUBMIT`，记录前后状态 |
| 审核通过 | `PENDING_X -> PENDING_{X+1}/COMPLETED` | 写入`APPROVE` |
| 审核退回 | `PENDING_X -> REJECTED`，并写`reject_review_level=X` | 写入`REJECT`，`opinion`与`reject_level`必填 |
| 重新提交 | `REJECTED -> 上一关`，必要时清空`reject_review_level` | 写入`RESUBMIT`，记录前后状态 |

建议：稿件状态更新与流水写入必须在同一事务提交。

## 7. MyBatis-Plus 映射建议

## 7.1 实体与表映射
- `MsManuscript` -> `ms_manuscript`
- `MsReviewRecord` -> `ms_review_record`

## 7.2 主键与通用字段
- 主键策略：`@TableId(type = IdType.AUTO)`。
- 时间字段：`created_at`、`updated_at`建议结合自动填充（MetaObjectHandler）。
- 枚举字段：`status`、`action`建议使用Java枚举 + TypeHandler（或统一字符串常量）。

## 7.3 Mapper建议
- 基础CRUD使用`BaseMapper<T>`。
- 列表分页、详情流水等查询按业务新增自定义SQL，避免复杂Wrapper可读性下降。
