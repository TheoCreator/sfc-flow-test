# 三审制稿件工作台 - 代码模块与接口设计（MVP）

本文基于前序文档，给出后端代码模块划分与接口定义，目标是让开发可直接按文档落地实现。

---

## 1. 设计目标

- 模块职责清晰，避免状态机规则分散。
- 接口定义统一，便于原生HTML/JS前端直接联调。
- 严格对齐三审流程：提交、通过、退回、重提、完成。

---

## 2. 后端代码模块设计（Spring Boot + MyBatis）

建议包结构（示例）：

```text
com.sfc.flowtest
├─ common                  # 通用响应、异常、枚举、工具
├─ manuscript              # 稿件领域
│  ├─ controller
│  ├─ service
│  ├─ mapper
│  ├─ entity
│  ├─ dto
│  └─ vo
├─ review                  # 审核与流水领域
│  ├─ controller
│  ├─ service
│  ├─ mapper
│  ├─ entity
│  ├─ dto
│  └─ vo
├─ publish                 # 审核完成后的第三方发布集成（Hook，与主事务解耦）
│  ├─ event                # 领域事件（如稿件完成）
│  ├─ hook                 # 第三方发布接口及模拟实现
│  └─ service              # 事件监听与 Hook 分发
└─ config                  # MyBatis、全局异常、跨域、日志配置
```

## 2.1 核心模块职责

| 模块 | 职责 |
|------|------|
| `common` | 统一响应结构、错误码、全局异常处理、状态/动作枚举 |
| `manuscript` | 稿件增改查、列表检索与分页、草稿保存、提交审核、重提入口 |
| `review` | 审核通过/退回、审核流水记录与查询 |
| `publish` | 审核完成（`COMPLETED`）后的第三方发布 Hook 预留与模拟对接 |
| `config` | MyBatis 配置、接口跨域策略等 |

## 2.2 关键类建议

| 类名（建议） | 说明 |
|--------------|------|
| `ManuscriptController` | 提供稿件列表、详情、创建、保存草稿、提交、重提接口 |
| `ReviewController` | 提供审核通过、退回、流水查询接口 |
| `ManuscriptService` | 稿件状态流转编排（提交、重提） |
| `ReviewService` | 审核动作处理（通过、退回） |
| `StateMachineService` | 统一封装状态迁移合法性校验 |
| `MsManuscriptMapper` | `ms_manuscript`数据访问 |
| `MsReviewRecordMapper` | `ms_review_record`数据访问 |
| `ManuscriptCompletedEvent` | 稿件进入完成态后触发的领域事件（供发布集成订阅） |
| `ThirdPartyPublishHook` | 第三方发布扩展点；默认可装配 `MockThirdPartyPublishHook` 打日志模拟 |
| `PublishHookDispatcher` | 事务提交后分发 Hook，并将结果摘要写入审核流水（`PUBLISH`） |

## 3. 接口总体约定

## 3.1 协议约定

- 协议：HTTP + JSON
- 路径前缀：`/api/v1`
- 字符集：`UTF-8`
- 时间格式：ISO8601或`yyyy-MM-dd HH:mm:ss.SSS`（统一即可）

## 3.2 统一响应结构

```json
{
  "code": "0",
  "message": "success",
  "data": {}
}
```

字段说明：
- `code`：`0`表示成功，非`0`表示业务或系统错误。
- `message`：结果说明。
- `data`：响应数据主体，失败场景可为`null`。

## 3.3 常用错误码建议

| 错误码 | 含义 |
|--------|------|
| `40001` | 参数校验失败 |
| `40002` | 稿件不存在 |
| `40003` | 当前状态不允许该操作 |
| `40004` | 退回意见不能为空 |
| `40005` | 退回关卡数据异常 |
| `50000` | 系统内部错误 |

## 4. 接口清单与详细设计

## 4.1 稿件模块接口

## 4.1.1 创建稿件
- 方法：`POST`
- 路径：`/api/v1/manuscripts`

请求体：
```json
{
  "title": "标题示例",
  "body": "正文示例"
}
```

处理规则：
- 新建稿件默认状态为`DRAFT`。
- 同步写入一条`SAVE_DRAFT`流水（可选但建议，便于全链路审计）。

## 4.1.2 稿件列表
- 方法：`GET`
- 路径：`/api/v1/manuscripts`
- 查询参数（可选）：
  - `status`：单状态筛选（枚举名，如 `PENDING_1ST`）
  - `statuses`：多状态筛选，**英文逗号分隔**（如 `PENDING_1ST,PENDING_2ND,PENDING_3RD`）；与 `status` 同时出现时，后端会合并为「去重后的多状态 IN 条件」
  - `keyword`：标题或正文**模糊检索**（`LIKE`，长度上限 100，超出返回 `40001`）
  - `pageNo`：页码，从 1 开始；与 `pageSize` 任一出现时进入分页模式（缺省按 `pageNo=1`、`pageSize=10`）
  - `pageSize`：每页条数，最大 100，超出返回 `40001`

返回字段（核心）：
- `id`、`title`、`status`、`updatedAt`

说明：
- 未传 `pageNo` 且未传 `pageSize` 时，返回**全量**匹配结果（数据量大时建议前端始终带分页参数）。
- 检索为 SQL 层 `LIKE` 实现，属 MVP 级「简单全文检索」，非搜索引擎方案。

## 4.1.3 稿件列表总数统计
- 方法：`GET`
- 路径：`/api/v1/manuscripts/count`
- 查询参数（可选）：与列表接口**相同的筛选条件**，不含分页：`status`、`statuses`、`keyword`

响应 `data` 结构：
```json
{
  "total": 123
}
```

用途：前端分页展示「第 X 页 / 共 Y 条」、菜单红点数量等，避免为统计再拉全量列表。

## 4.1.4 稿件详情
- 方法：`GET`
- 路径：`/api/v1/manuscripts/{id}`

返回字段（核心）：
- `id`、`title`、`body`、`status`、`rejectReviewLevel`、`createdAt`、`updatedAt`

## 4.1.5 保存草稿
- 方法：`PUT`
- 路径：`/api/v1/manuscripts/{id}/draft`

请求体：
```json
{
  "title": "更新后的标题",
  "body": "更新后的正文"
}
```

处理规则：
- 仅允许`DRAFT`或`REJECTED`状态执行编辑保存。
- `COMPLETED`状态禁止编辑。
- 写入`SAVE_DRAFT`流水。

## 4.1.6 提交审核
- 方法：`POST`
- 路径：`/api/v1/manuscripts/{id}/submit`

处理规则：
- 仅允许`DRAFT`提交。
- 状态变更：`DRAFT -> PENDING_1ST`
- 写入`SUBMIT`流水。

## 4.1.7 重新提交（退回后）
- 方法：`POST`
- 路径：`/api/v1/manuscripts/{id}/resubmit`

处理规则：
- 仅允许`REJECTED`状态执行。
- 根据`reject_review_level`计算目标状态：
  - `1 -> DRAFT`
  - `2 -> PENDING_1ST`
  - `3 -> PENDING_2ND`
- 写入`RESUBMIT`流水，并记录前后状态。

---

## 4.2 审核模块接口

## 4.2.1 审核通过
- 方法：`POST`
- 路径：`/api/v1/manuscripts/{id}/approve`

处理规则：
- 仅允许在`PENDING_1ST`、`PENDING_2ND`、`PENDING_3RD`执行。
- 状态迁移：
  - `PENDING_1ST -> PENDING_2ND`
  - `PENDING_2ND -> PENDING_3RD`
  - `PENDING_3RD -> COMPLETED`
- 写入`APPROVE`流水。
- 当下一状态为`COMPLETED`时，在**同一事务内**发布 `ManuscriptCompletedEvent`；第三方发布实际执行在**事务提交之后**（见下文「发布集成」），避免外部失败导致审核事务回滚。

## 4.2.2 审核退回
- 方法：`POST`
- 路径：`/api/v1/manuscripts/{id}/reject`

请求体：
```json
{
  "opinion": "请补充事实来源与数据出处"
}
```

处理规则：
- 仅允许在`PENDING_1ST`、`PENDING_2ND`、`PENDING_3RD`执行。
- `opinion`必填。
- 状态统一变更为`REJECTED`，并保存当前关卡到`reject_review_level`。
- 写入`REJECT`流水，包含`opinion`和`reject_level`。

## 4.2.3 审核流水查询
- 方法：`GET`
- 路径：`/api/v1/manuscripts/{id}/records`

返回字段（核心）：
- `action`、`fromStatus`、`toStatus`、`opinion`、`rejectLevel`、`operatorName`、`createdAt`

其中 `action` 在 MVP 扩展中包含：`PUBLISH`（第三方发布 Hook 执行结果摘要，见下节）。

---

## 4.3 发布集成（第三方 Hook，与主流程解耦）

### 4.3.1 设计目的
- 审核主链路（状态 + 流水）保持**短事务**、可回滚。
- 对接外部平台（CMS、内容中台等）属于**出站集成**：失败重试、超时、幂等等不应阻塞审核落库。

### 4.3.2 触发时机
- 三审最后一次通过，稿件状态变为 `COMPLETED` 后触发。

### 4.3.3 扩展方式
- 定义接口 `ThirdPartyPublishHook`：`name()`、`publish(ManuscriptDetailVO manuscript)`。
- 默认实现 `MockThirdPartyPublishHook`：仅打日志，模拟外部 HTTP/SDK 调用。
- `PublishHookDispatcher` 使用 `@TransactionalEventListener(phase = AFTER_COMMIT)` 在事务提交后依次调用所有已注册的 `ThirdPartyPublishHook` Bean。

### 4.3.4 可观测性（流水）
- 每个 Hook 执行结束（成功或捕获异常）后，追加一条审核流水，`action = PUBLISH`，`fromStatus/toStatus` 均为 `COMPLETED`，`opinion` 中写明 `hook=<名称>` 及成功/失败摘要，便于联调与审计。

### 4.3.5 配置项（`application.yml`）
- `sfc.publish.enabled`：是否启用发布 Hook（默认 `true`）。
- `sfc.publish.mock-endpoint`：模拟第三方地址字符串，仅用于日志展示（默认 `mock://external-platform/manuscript`）。

## 5. 关键业务流程（接口级）

## 5.1 正向通过链路
1. 创建稿件（`POST /manuscripts`）
2. 提交审核（`POST /manuscripts/{id}/submit`）
3. 三次审核通过（`POST /manuscripts/{id}/approve`）
4. 状态进入`COMPLETED`

## 5.2 退回重提链路
1. 待审态执行退回（`POST /manuscripts/{id}/reject`）
2. 稿件进入`REJECTED`并记录`reject_review_level`
3. 编辑草稿（`PUT /manuscripts/{id}/draft`）
4. 重新提交（`POST /manuscripts/{id}/resubmit`）回到对应节点

---

## 6. 事务与一致性要求

- 以下操作必须单事务：
  - 状态变更 + 流水记录写入
- 任一步骤失败时整体回滚，防止出现：
  - 状态已更新但流水缺失
  - 流水已记录但状态未更新
- **第三方发布**不得在审核主事务内执行远程调用；采用「主事务提交后再触发 Hook」的方式，发布失败时通过 `PUBLISH` 流水与错误日志体现，不回滚已完成审核结果。

## 7. 原生HTML/JS前端联调要点

- 页面通过`fetch`调用以上REST接口。
- 按状态控制按钮显隐（提交/通过/退回/重提）。
- 退回操作前端需校验`opinion`非空；后端必须二次校验。
- 详情页同时拉取稿件详情与流水列表，保证“当前状态 + 历史轨迹”同屏展示。
- 列表页建议并行请求 `GET /api/v1/manuscripts`（分页数据）与 `GET /api/v1/manuscripts/count`（同条件 `total`），用于「第 X 页 / 共 Y 条」与菜单角标统计。
- 静态页已实现侧栏模块（稿件列表 / 我的审批 / 已完成）与操作列按钮样式，仍以接口返回为准做显隐与刷新。

## 8. 开发规范（遵循阿里Java开发手册）

本项目编码规范默认遵循《阿里巴巴Java开发手册》强制项与推荐项，以下为本项目落地约束。

## 8.1 命名与结构规范
- 包名、方法名、参数名统一小写驼峰；类名使用大驼峰；常量全大写下划线分隔。
- `Controller/Service/Mapper/Entity/DTO/VO`分层命名清晰，禁止跨层直接调用。
- 禁止使用拼音缩写、单字符无语义命名（循环变量等临时变量除外）。

## 8.2 Controller层规范
- `Controller`只做参数接收、基础校验、返回封装，不写核心业务逻辑。
- 所有写操作接口必须声明请求DTO，禁止直接暴露Entity作为入参。
- 统一返回结构（`code/message/data`），禁止直接返回原始异常堆栈。

## 8.3 Service层与事务规范
- 业务规则（状态机流转、退回/重提判定）必须收敛在`Service`层。
- 涉及“状态更新 + 流水写入”的方法必须使用事务，保证原子性。
- 同一事务中禁止远程调用外部系统，避免长事务和不确定回滚。

## 8.4 异常与错误码规范
- 使用业务异常（如`BizException`）承载业务错误，不抛`Exception`泛型。
- 参数校验失败、状态不合法、数据不存在等必须映射明确错误码。
- 全局异常处理器统一转换响应，避免控制器层重复`try-catch`。

## 8.5 日志规范
- 使用参数化日志：`log.info("manuscriptId={}, action={}", id, action)`，禁止字符串拼接。
- 错误日志必须包含关键上下文：稿件ID、动作、前后状态、错误码。
- 禁止打印密码、token、连接串等敏感信息。

## 8.6 数据库与MyBatis-Plus规范
- 表名/字段名统一下划线命名，实体属性驼峰命名。
- 主键统一`BIGINT`自增，实体主键策略与表设计保持一致。
- 禁止`SELECT *`，查询字段按需选择；批量操作需显式限制范围。
- Mapper层只做数据访问；复杂业务判断不得下沉到SQL脚本。

## 8.7 接口与安全基础规范
- `GET`只做查询，`POST/PUT`用于状态变更，语义与HTTP方法一致。
- 入参必须进行空值、长度、枚举合法性校验（如`opinion`必填）。
- 生产配置不得提交明文敏感信息；使用环境变量注入。

## 8.8 注释与文档规范
- 类、核心方法、复杂分支需写“为什么这样做”的注释，而非复述代码。
- 变更接口或状态机规则时，必须同步更新`doc/01`、`doc/03`、`doc/04`。
- 关键流程（提交、退回、重提）建议补充单元测试用例名与场景注释。
